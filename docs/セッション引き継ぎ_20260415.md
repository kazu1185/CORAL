## 1. 予約統合機能のコード再検証

開発が遠回りしたため、最終的なコードに問題がないか全面検証を実施。

### 検出・修正した問題

#### 1-1. UNIQUE KEY定義の不一致（修正済み）
- **ファイル**: `backend/migrations/010_reservation_merge.sql`
- **問題**: `reservation_sources` のUNIQUE KEYがマイグレーション010では `(channel, reservation_no)` だったが、schema.sqlでは `(reservation_id, channel, reservation_no)`
- **影響**: 分割→再統合フローでUNIQUE制約違反の可能性
- **対応**: 010を修正（本番DBは011で既に修正済み）

#### 1-2. split時のreservation_noがNULL連結で `-S` になる（修正済み）
- **ファイル**: `backend/src/Controllers/ReservationController.php:1216`
- **問題**: 統合予約（reservation_no=NULL）を分割すると `-S` になる
- **対応**: `($reservation['reservation_no'] ?? $reservation['id']) . '-S'` に変更

#### 1-3. TLインポートでmerged子予約にdirectマッチしてしまう（修正済み・重要）
- **ファイル**: `backend/src/Services/TlImportService.php:711`
- **問題**: `findExistingReservation()` がmerged状態の子予約にもdirectマッチし、TL変更通知が親ではなく子予約を直接変更してしまう
- **対応**: `$result['status'] === 'merged'` の場合はdirect扱いせず、Step 2（source経由検索）に進むよう修正

#### 1-4. merge_alertのevent_atが過去日になる（修正済み）
- **ファイル**: `backend/src/Services/TlImportService.php:1495`
- **問題**: `recordMergeAlert()` が `booked_at`（OTA予約日=過去日）を event_at に使用しており、ダッシュボードの7日以内フィルタに引っかからない
- **対応**: `null`（=NOW()）に変更

### 検証結果: 問題なし
- merge（統合）: 楽観ロック、日程連続性、二重統合防止、sources/charges付替え — すべて正常
- split（分割）: sources/chargesの日付ベース振り分け — 正常
- unmergeSource（統合解除）: 中日解除→完全解除の判断 — 正常
- TLインポート連携: 金額変更の自動反映、日程変更のmerge_alert — 正常
- デッドコード: なし

## 2. merge_alert承認機能（新規実装）

### 背景
TLから統合予約内の子予約に日程変更・取消通知が来た場合、`merge_alert` イベントを記録するが、スタッフが対処する手段がなかった。

### 実装した機能

#### 方針
統合予約内の子に変更があった場合、**統合を一旦すべて分解**するシンプルな方式を採用。部分的な再統合は複雑すぎるため、分解後にスタッフが必要に応じて手動で再統合する。

#### 処理フロー
1. ダッシュボードにmerge_alertアラート表示（赤、`link_off`アイコン）
2. アラートクリック → 予約詳細画面に遷移
3. 予約詳細画面に赤いアラートバー + 「確認して分解」ボタン
4. ボタン押下 → 確認ダイアログ（変更内容 + 分解後の予約一覧を表示）
5. 「分解を実行」→ 統合全解除 + 変更通知の反映（日程変更 or キャンセル）

#### 修正ファイル

| ファイル | 内容 |
|---------|------|
| `backend/config/routes.php` | `POST /api/v1/reservations/:id/resolve-merge-alert` 追加 |
| `backend/src/Controllers/ReservationController.php` | `resolveMergeAlert()` 新メソッド（全解除 + 日程変更/取消の反映） |
| `backend/src/Controllers/ReservationController.php` | `show()` に `pending_merge_alerts` 配列を追加 |
| `backend/src/Controllers/DashboardController.php` | `getAlerts()` にmerge_alertアラート追加 + 処理済み除外フィルタ |
| `frontend/src/pages/ReservationDetailPage.jsx` | 赤アラートバー + `MergeAlertResolveDialog` コンポーネント |
| `frontend/src/pages/ReservationDetailPage.jsx` | タイムラインのdetail表示を人間可読形式に整形（`formatEventDetail()`） |
| `frontend/src/pages/ReservationDetailPage.jsx` | ホバー→クリック式ポップオーバーに変更（`TimelineDetailPopover`）|
| `frontend/src/pages/ReservationDetailPage.css` | 赤アラート・分解ダイアログ・ポップオーバーのスタイル |
| `frontend/src/pages/DashboardPage.jsx` | merge_alert用アイコン `link_off` 適用 |

#### resolveMergeAlert の処理詳細
1. 楽観ロック確認
2. merge_alertイベントからalert_type/channel/reservation_noを取得
3. 全sources取得 → 全解除（全子予約をconfirmedに復元、charges付替え、親をOTA予約に復元）
4. date_changeの場合: 該当予約の日程を変更後の値に更新 + 範囲外chargesを論理削除 + amount再計算
5. cancellationの場合: 該当予約をcancelledに変更
6. merge_alertのsummaryに「→ 統合解除で対応済み」追記
7. `merge_alert_resolved` イベント記録 + 操作ログ

#### pending_merge_alerts の判定ロジック
- `event_type = 'merge_alert'` かつ `summary NOT LIKE '%対応済み%'`
