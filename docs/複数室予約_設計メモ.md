# 複数室予約（団体予約）設計メモ

## 2026-04-13 実装

### 背景

Agoda/Booking.comで1電文にN室含まれる予約がある（TotalRoomCount > 1）。
従来は1予約=1室の前提で、全室分の金額が1レコードにまとまっていた。

### 方式: 親子分割 + reservation_sources 流用

1電文（N室）→ **親1件 + 子N件** の独立予約に分割。

- 親予約: OTA予約そのもの（channel, reservation_no保持, status=`group_parent`）
- 子予約: 各部屋（独立CI/CO/精算, `parent_reservation_id` で親を参照）

### reservation_sources との関係

通常のTL取込は `parent_reservation_id` で完結する。
reservation_sources が活きるのは以下のケース:

- 子予約がスタッフにより**別の予約と統合**された場合
  → 子のOTA情報が reservation_sources に入る
  → TL通知 → 親にヒット → 子を探す → 子がmerged → reservation_sources経由で統合先を更新

### スキーマ変更（マイグレーション 014）

reservations テーブルに3カラム追加:

| カラム | 型 | 説明 |
|--------|-----|------|
| parent_reservation_id | INT NULL | 子→親のFK |
| room_count | INT DEFAULT 1 | 室数（親で使用） |
| room_index | INT NULL | 親内の室番号（1-based、子のみ） |

status ENUM に `group_parent` を追加。

### TL取込フロー

#### 新規（room_count > 1）

1. XMLパース → `extractDailyRatesByRoom()` で部屋別料金取得
2. 親予約INSERT（status=`group_parent`, 合計金額）
3. 子予約N件INSERT（各部屋の料金・タイプ）+ charges生成
4. guest_linksで子予約を連結（LinkedBar表示用）

#### 取消

1. `findExistingReservation(reservation_no)` → 親にヒット
2. 親の`room_count > 1` を検出
3. 子もCI済みチェック → 全件キャンセル + アサイン物理削除

#### 変更

| 条件 | 処理 |
|------|------|
| 室数変更あり | merge_alert（スタッフ判断） |
| 日程変更あり | merge_alert（スタッフ判断） |
| 金額のみ変更 | 各子の charges 自動更新 |

### 表示

- 予約一覧: `group_parent` は除外（子予約のみ表示）
- 予約詳細: 子は親情報+兄弟一覧を表示、親は子一覧を表示
- アサインボード: `group_parent` は未アサインから除外
- guest_linksによるLinkedBar表示はそのまま動く

### 実データ件数（引き継ぎ書より）

- Agoda: 最大5室（SW×5）、2〜4室も多数
- Booking.com: 部屋タイプ混在あり（TW+SW、TW+LR）→ 4件確認済み
- 変更通知1件、取消通知4件

### XML構造

`RoomAndGuestList` が「日付×室数」分並ぶ:
- 同一日付のN番目 = N番目の部屋
- 各エントリに `RoomTypeCode` + `TotalPerRoomRate`
- Booking.comは `PerRoomPaxCount` あり、Agodaは1室目に全員分
