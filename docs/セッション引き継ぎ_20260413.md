## 未実装: 団体予約（複数室予約）対応

### 背景

Agoda/Booking.com等で1電文に複数室が含まれる予約がある。現在PMSは1予約=1室の前提で、複数室予約は1つの予約レコードに全室分の金額・人数がまとまって入る。

実データ:
- Agoda: 最大5室（SW×5）、2〜4室も多数
- Booking.com: **部屋タイプ混在あり**（TW+SW、TW+LR）→ 4件確認済み
- 変更通知1件、取消通知4件の実データあり

### 電文構造の調査結果

TLリンカーンのフォーマットはOTA共通。`RoomAndGuestList` が「日付×室数」分並ぶ構造:
- 同一日付のN番目のエントリ = N番目の部屋
- 各エントリに `RoomTypeCode`（部屋タイプ）と `TotalPerRoomRate`（料金）が独立
- **部屋ごとの料金パースは可能**（Agoda/Booking共通ロジックで対応できる）
- 人数: Booking.comは部屋ごとの `PerRoomPaxCount` あり、Agodaは1室目に全員分が入り他は空

パーサー `TlXmlParser::extractDailyRatesByRoom()` は実装・検証済み（巻き戻したが、コードは引き継ぎ可能）:
```php
// 返却値: ['rooms' => [0 => ['room_type'=>'TW', 'rates'=>[...]], 1 => ...], 'room_types' => ['TW','SW']]
```

### ハイブリッド方式（実装→巻き戻し）

1予約レコードにroom_countを持たせ、room_assignmentsを拡張する方式で実装したが、以下の問題で巻き戻し:
- 個別CI/CO: アサイン単位のステータス管理が必要 → 二重管理
- 個別精算: chargesにassignment_id/room_indexが必要 → 既存ロジック全面改修
- 売上明細表示: 部屋ごとフィルタが必要 → UI複雑化
- **本質的に「1レコードに5つの独立情報を詰め込んでいるだけ」**

### 次セッションで決定すべき: 分割方式の詳細設計

**基本方針: 1電文（N室）→ N件の独立予約レコードに分割**
- 各予約が独立した1室予約 → 既存のCI/CO/精算/アサインが全てそのまま動く

**3つの関連付け方式の比較:**

#### 方式1: guest_links（既存テーブル）

| 利点 | 懸念 |
|------|------|
| 既にTL複数室予約（別電文パターン）で使用中 | TL変更/取消: `findExistingReservation()` が `LIMIT 1` → 全件取得に改修必要 |
| `LinkedBar` でアサインボードにグループ表示済み | 部屋数減（5→3）の変更通知: どの2件を削除するか判断ロジック必要 |
| 追加テーブル不要 | guest_linksは予約番号を持たない（逆引き不可） |
| | 既存用途（別電文の複数室連結）と混在する |

#### 方式2: reservations に group_reservation_no 追加

| 利点 | 懸念 |
|------|------|
| シンプル。追加テーブル不要 | 各子予約の `reservation_no` をどうするか（同じ値→複数ヒット、枝番→LIKE必要） |
| `WHERE group_reservation_no = X` で全件取得可能 | `reservation_no` 参照箇所が全面影響 |
| | 統合予約（`reservation_no = NULL`）との整合性が曖昧 |

#### 方式3: reservation_sources を流用

| 利点 | 懸念 |
|------|------|
| `findReservationViaSource()` が既に動いている | 統合とは方向が逆（統合: N→1、複数室: 1→N） |
| 金額変更の自動更新ロジックも実装済み | `LIMIT 1` → 同じreservation_noで5件のsourceがある場合の挙動 |
| 日程変更/取消は merge_alert でスタッフに委ねる設計あり | match_via='source' の処理が統合用なので区別が必要 |
| UNIQUE制約 `(reservation_id, channel, reservation_no)` は子予約のIDが異なるのでOK | |

### 親あり vs 親なし

**親なし（N件の独立予約のみ）:**
- TL取消 → 5件全部キャンセル → OK
- TL変更（5室→3室）→ **どの2件を削除するか判断不能**
- 一括精算 → 5件横断の合計が必要

**親あり（親1件 + 子N件）:**
- 親予約 = OTA予約そのもの（channel=agoda, reservation_no=1658136180, 合計金額）
- 子予約 = 各部屋（独立CI/CO/精算）
- TL取消 → 親にヒット → 子も全部取消（明快）
- TL変更（5室→3室）→ 親のroom_count/金額更新 → 余剰の子をキャンセル
- 一括精算 → 親予約IDで全体合計を参照
- **予約統合と同じ構造の逆パターン**

| | 予約統合 | 複数室分割 |
|--|---------|---------|
| 入力 | N件のOTA予約 | 1件のOTA電文（N室） |
| 親予約 | 統合先（channel=NULL） | OTA予約そのもの（channel=agoda） |
| 子予約 | merged化 | 独立予約（各自CI/CO可能） |
| reservation_sources | 元OTA番号を保持 | 親→子の紐付け |
