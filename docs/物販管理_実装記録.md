### DB変更（022）

- **products**（新規）: 商品名・カテゴリ・税込価格・税率(8/10)・sort_order・is_active
- **product_sales**（新規）: 販売記録の本体。商品名/単価/税率をスナップショット保持
- **reservation_charges**: `charge_type` に `'goods'` 追加、`tax_rate TINYINT NULL` 追加
- **document_items**: `tax_rate TINYINT NULL`、`sale_id INT NULL` 追加
- **permissions**: `product_sales.view` / `product_sales.create`（category=`物販`, sort_order 33/34）、`master.products`（category=`設定`, sort_order 54）
- **role_permissions**: 4ロール×3権限=12行。product_sales系はフロント系3ロールに付与（`receipt.issue` と同じ）、master.products は admin/front_manager のみ（`master.plans` と同じ）

### 設計判断

- **DB名は `DATABASE()` から取得**（017等と同じ流儀）。`TABLE_SCHEMA='pms_db'` ハードコードの旧マイグレーションもあるが、本番DB名が異なった場合に静かに失敗するため新規は DATABASE() を使う
- **charge_type の ENUM 追加は「COLUMN_TYPE LIKE '%goods%'」で存在チェック**してから MODIFY。再実行しても安全にした
- **`reservation_charges.tax_rate` / `document_items.tax_rate` は NULL 許容**。既存行を一括更新せず NULL = 10%扱いとする（TL由来の既存明細に触らないため）
- **税率の選択肢 8/10 のバリデーションはコードに書いてよい**と判断。禁止されているのは「金額計算に税率を埋め込むこと」であり、入力値の妥当性検証は別物。計算は常に `products.tax_rate` / `product_sales.tax_rate` を参照する
- **商品マスタの GET は `reservation.view` 権限**（プラン・決済方法と同じ）。販売画面からも商品一覧を読むため
- **一覧APIは `all=1` で非アクティブ含む**。販売画面は無効商品を出さないためデフォルトは active のみ
- POST時のみ price / tax_rate を必須にした。PUT は `useMasterCrud` の is_active トグル（部分更新）が通る必要があるため、送られた項目だけ検証する

### 動作確認結果

- マイグレーション適用OK（日本語カラムコメント・権限名の文字化けなし。`--default-character-set=utf8mb4` 使用）

- 無効化した商品が `all=1` なしの一覧APIから除外されることを確認（販売画面で売れない商品が出ない）

### 設計判断

- **商品情報はフロントから受け取らずサーバーでマスタから読み直す**（価格の改ざん防止）。販売日もサーバーの当日を使う（端末の時計ずれ対策）
- **即売と部屋付けは排他**。両方指定・どちらも未指定はエラー。部屋付けは `checked_in` の予約のみ許可
- **ProductPicker は確定ボタンを `actions` で差し込む設計**にした。確定手段が画面ごとに違う（物販ページ＝支払方法選択＋部屋付け、予約詳細＝部屋付けのみ）ため、カートの状態管理だけを共通化した
- **決済方法はボタン羅列ではなくセレクト**。OTA決済を含め11件あり、ボタンだと縦に伸びて商品グリッドが押しにくくなるため
- **予約一覧APIに `updated_at` を追加**。物販ページの部屋付けで楽観ロック（規約 #16）に必要。列が増えるだけなので他画面への影響なし
- 内税計算式はフロント（`utils/tax.js`）とバック（`ProductSaleController::calcTaxAmount()`）で同一。**税率は引数で受け取り、式には埋め込まない**
- **`recalcReservationAmount()` は ReservationController の `recalcAmountFromCharges()` と同じ条件式**にした。private 同士で共有できないため意図的に同じ流儀で書いている（条件を変える場合は両方直すこと）

### 動作確認結果

- 即売（複数商品）→ 履歴表示 → 取消 の一連が動作。取消済みは打ち消し線で残る（物理DELETEなし）
- 内税計算の検算: ¥800(10%)→¥73、¥650(8%)→¥49、¥1,950(8%)→¥145 いずれも一致
- バリデーション: 両方指定 / どちらも未指定 / 数量0 / 無効な支払方法 / 二重取消 すべて400系で弾く
- 楽観ロック: 古い `updated_at` を送ると409

### 設計判断

- **goods 行は明細テーブルから編集・削除できないようにした**（鍵アイコン＋ツールチップ）。`product_sales` と1対1で対応するため、片方だけ変えると売上集計が食い違う。取消は物販ページの履歴から行い、`product_sales` と `reservation_charges` を連動して cancelled にする
- **手動の明細行追加のセレクトから goods を除外**（`MANUAL_ADD_EXCLUDED_TYPES`）。product_sales の無い goods 行を作らせないため。ただし編集時のセレクトには残す（既存 goods 行の種別が消えないように）
- 部屋付けの明細は `description = 商品名 ×N`、`date = 販売日`、`tax_rate` を設定

### 動作確認結果

- 予約詳細から物販追加 → 明細に goods 行が出る → 請求額・未収が更新される
- 物販ページからの部屋付けも同じ結果（予約971で amount 51,702 → 51,922 → 52,072）
- 取消すると `reservation_charges` も cancelled になり amount が戻る

## フェーズ4: 即売領収書＋税率区分記載（2026-07-22 完了）

### 追加・変更したファイル

| ファイル | 内容 |
|---------|------|
| `backend/src/Services/DocumentService.php` | goods を領収書対象に追加、document_items に tax_rate、`issueSalesReceipt()` 新規、再発行で tax_rate/sale_id をコピー |
| `backend/src/Services/ReceiptPdfService.php` | 税率区分記載（`summarizeByTaxRate()`）、お支払方法欄 |
| `backend/src/Controllers/DocumentController.php` | `issueSalesReceipt()`、show で payment_method_name を取得 |
| `backend/config/routes.php` | `/documents/sales-receipt`（`:id` より前に配置） |
| `frontend/src/pages/ProductSalesPage.jsx` / `.css` | 履歴からの領収書発行（会計単位）、発行済み表示 |
| `backend/migrations/022_product_sales.sql` | `product_sales.sale_group_id` を追加 |

### 設計判断

- **1会計＝1枚の領収書**。1回の販売でも商品ごとに product_sales 行が分かれるため、`sale_group_id`（その会計の先頭行のid）で束ねる。履歴の「領収書」ボタンは会計の先頭行にだけ出し、その会計の全 sale_id をまとめて送る
- **税率区分は「軽減税率の明細が無ければ従来と同じ1行」**になるように実装した。標準税率(10%)の対象額は「合計 − 宿泊税 − 軽減税率分」として逆算する。宿泊料は明細に税額を持たない運用のため、既存の考え方をそのまま残している。結果、物販導入前の領収書のPDFは見た目が変わらない
- **二重発行の防止**: 同じ sale_id で有効な領収書が既にあれば400で弾く（再発行は既存の再発行フローを使う）
- **即売領収書の `reservation_id` は NULL**。`reservation_events` は予約に紐づく履歴なので即売では記録しない（FK制約もある）
- 但し書きの既定は即売のみ「品代として」（宿泊は従来どおり「宿泊代として」）
- **お支払方法をPDFに追加**（ユーザー要望）。「但 〜」の行の右端に表示。`payment_method_id` が未設定の帳票では出さないので既存領収書の見た目は変わらない

### 動作確認結果

- 8%/10%混在の宿泊領収書: 消費税（10%）¥21・消費税（8%）¥12、注記「※10%対象 ¥51,922（税¥21）」「※8%対象 ¥150（税¥12）」。10%対象＋8%対象＝合計と一致
- 物販なしの宿泊領収書: 注記は「※10%対象: ¥121,500（税: ¥0）」の1行のみ＝従来と同じ
- 即売領収書（3品目1会計）: 1枚に3明細、税率区分も正しい
- 二重発行・部屋付け分の即売領収書化はエラーで弾かれる

### 設計判断

- **集計元は product_sales**。部屋付け分は `reservation_charges` にも計上されているため、宿泊売上のレポートと足し合わせると二重計上になる。この注意はコントローラーのコメントと画面の注記の両方に明記した
- 支払方法別は即売のみ（部屋付けはCO精算時に決まるため、販売時点では確定していない）
- 商品別の集計キーは `product_id + product_name + tax_rate`（販売時のスナップショット）。マスタで改名・改定しても過去の集計が変わらない

### 実施できなかった項目

- 指示書 2-5 の「既存 daily / monthly に即売分の売上を追加」は、`ReportController::daily()` / `monthly()` が未実装（501を返すだけ）のため対象がなかった
- 実際に稼働している集計は「収入実績・予測表PDF」（`IncomeReportService`）で、こちらは `charge_type IN ('room','addon','discount')` を売上対象としており **goods を含んでいない**。物販を宿泊の収入表に含めるかは帳票の意味が変わる判断のため、ユーザー確認事項として保留

### 動作確認結果

- 期間集計が product_sales と一致（即売¥5,120＋部屋付け¥370＝合計¥5,490）
- 税率別・支払方法別・商品別の内訳も一致
