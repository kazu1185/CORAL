# テストデータ投入 実装指示書

Claude Code向け。この指示書に従ってseed.sqlを生成すること。

---

## 0. 前提

- DB名: pms_db（27テーブル作成済み）
- seed.sql は database/ ディレクトリに配置
- `mysql -u pms_user -p pms_db < database/seed.sql` で何度でも投入可能にする
- 先頭にDELETEまたはTRUNCATE文を入れて、再実行で冪等になるようにすること
- テストデータ設計書（docs/テストデータ設計書.md）のパターンを網羅すること
- 外部キー制約を考慮した投入順序にすること

---

## 1. 投入順序

```
① ALTER TABLE（不足カラム追加）
② マスタデータ（room_types, rooms, plans, accommodation_tax_rules, payment_methods, system_settings）
③ スタッフ・権限データ（staff, permissions, role_permissions, device_tokens）
④ ゲストデータ（guests, guest_aliases）
⑤ 予約データ（reservations, reservation_charges, room_assignments）
⑥ 連結予約データ（guest_links）
⑦ 法人データ（corporate_clients）
```

---

## 2. ALTER TABLE（不足カラム追加）

staffテーブルにログインロック用カラムが未作成の可能性がある。存在しなければ追加する。

```sql
-- ログインロック用カラム（存在チェック付き）
ALTER TABLE staff ADD COLUMN IF NOT EXISTS login_fail_count INT NOT NULL DEFAULT 0;
ALTER TABLE staff ADD COLUMN IF NOT EXISTS last_login_fail_at DATETIME NULL;

-- reservationsテーブルのCI/CO実績カラム（存在チェック付き）
ALTER TABLE reservations ADD COLUMN IF NOT EXISTS actual_checkin_at DATETIME NULL;
ALTER TABLE reservations ADD COLUMN IF NOT EXISTS actual_checkout_at DATETIME NULL;
ALTER TABLE reservations ADD COLUMN IF NOT EXISTS payment_method VARCHAR(50) NULL;
ALTER TABLE reservations ADD COLUMN IF NOT EXISTS reservation_notes TEXT NULL;

-- rooms テーブルの清掃ステータスカラム
ALTER TABLE rooms ADD COLUMN IF NOT EXISTS housekeeping_status ENUM('clean','cleaning','inspecting','dirty') NOT NULL DEFAULT 'clean';
ALTER TABLE rooms ADD COLUMN IF NOT EXISTS sort_order INT NOT NULL DEFAULT 0;
```

**注意: MySQL 8.4 では `ADD COLUMN IF NOT EXISTS` が使えない場合がある。その場合はストアドプロシージャか、エラーを無視する方式（Source コマンド実行時に --force オプション）で対処すること。代替として、カラム存在確認クエリを使ってもよい。**

## 5. ゲストデータ

### 5-1. guests（80名前後）

以下のパターンを含めること:

**日本人ゲスト（50名）:**
- 漢字・カナ・ローマ字すべて入力済み
- visit_count が 1〜8 の範囲でバラつかせる
- VIPフラグ付き: 5名程度
- guest_notes 付き: 10名程度（アレルギー、高層階希望、VIPフルーツ等）

**外国人ゲスト（15名）:**
- ローマ字のみ（漢字・カナはNULL）
- country_code: US, TW, KR, CN, AU, GB, TH, SG, FR, DE 等

**日本人だが海外OTA経由（10名）:**
- ローマ字のみ入力（カナ変換対象）
- country_code: JP

**特殊ケース:**
- 旧姓あり: 3名（guest_aliasesにも登録）
- 同姓同名の別人: 「田中 太郎」が2名
- マージ済みゲスト: 1名（status='merged', merged_into_guest_id セット済み）

### 5-2. guest_aliases（旧姓 — 3件）

旧姓ゲストに対応するalias。alias_type='maiden'。

## 6. 予約データ

### 6-1. データ期間

基準日: 2026-04-10（本日）を中心に前後2ヶ月分。

- 過去分（3/1〜4/9）: checked_out 済みの予約 60件程度
- 当日（4/10）: CI予定 5件、CO予定 4件、滞在中 10件
- 未来分（4/11〜5/31）: confirmed の予約 80件程度
- キャンセル済み: 10件
- ノーショー: 5件

**P-02: 連結予約の穴（2セット）**
- セット①: 10泊連結の6・7泊目がキャンセル
  - 5件の連結予約のうち、3件目がstatus='cancelled'
  - guest_linksのsequence=3がstatus='gap'

**P-03: 途中退室（3件）**
- 4泊予約の2泊目でCO。actual_checkout_atが入っている

**P-04: 部屋移動（3件）**
- room_assignments が2レコード（旧室released + 新室active）

**P-05: ノーショー（5件）**
- status='no_show'、reservation_charges に no_show_fee

**R-02: 高額予約（3件）**
- 1人1泊 100,000円超（宿泊税上限テスト）

**R-03: 低額予約（3件）**
- 1人1泊 1,000円未満（宿泊税0円テスト）

**K-01: 子供あり・child_amount入力済み（10件）**
- adult_count=2, child_count=1, child_amount=5000 等

**K-02: 子供あり・child_amount未入力（5件）**
- adult_count=2, child_count=1, child_amount=NULL

**D-02: OTA事前決済（15件）**

**D-03: Booking事前決済（10件）**
- Booking予約（payment_method='ota_prepaid'）

**D-04: 法人予約・月末締め（12件）**
- corporate_client_id 紐付き。3社分

**D-05: 法人予約・都度（8件）**
- corporate_client_id 紐付き。2社分

### 6-5. reservation_charges（売上明細）

全予約に対して1泊1行で生成。

- type='room' が基本
- ノーショーには type='no_show_fee'（通常は初泊分の料金）
- 途中退室には type='cancel_fee' または status='waived'
- 料金は room_types の default_rate ± ランダム変動（繁忙期は+20〜50%、閑散期は-10〜20%）

### 6-6. room_assignments（部屋アサイン）

- checked_out の予約: status='released'
- checked_in の予約: status='active'
- confirmed の予約: 8割はアサイン済み、2割は未アサイン（未アサインテスト用）
- 部屋移動ケース: 旧アサインreleased + 新アサインactive

## 8. 生成ルール

- 日本人名は実在しそうな一般的な名前を使うこと（テスト感のある「テスト太郎」等は避ける）
- 料金は現実的な範囲（シングル 6,000〜10,000、ツイン 12,000〜18,000、デラックス 18,000〜28,000、スイート 30,000〜50,000）
- 予約番号はOTAプレフィックス + 5桁連番で、重複なし
- 日付の分布は均一ではなく、金〜日は密に、火〜木は疎に（繁閑差）
- 外部キー制約を満たすように、参照先テーブルを先にINSERTすること
- seed.sqlの先頭で SET FOREIGN_KEY_CHECKS = 0; 末尾で SET FOREIGN_KEY_CHECKS = 1; を入れて安全にTRUNCATEできるようにする

## 10. 注意事項

- bcryptハッシュは固定値を使うこと（PHPで事前に `password_hash('1234', PASSWORD_DEFAULT)` を実行して得た値）
- DELETEではなくTRUNCATEを使用（AUTO_INCREMENTリセットのため）
- テストデータは開発・デバッグ用であり、本番では使用しない
- 170件全てを手書きする必要はない。パターンを網羅できる代表的なデータを作り、残りは連番で水増しすること
- ファイルサイズが巨大（数千行）になることが予想されるが、問題ない
