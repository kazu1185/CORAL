# コーディング規約・注意事項

開発中に判明した注意事項・約束事をまとめる。新しい知見が出たら追記すること。

---

## 1. コメント

- **なぜそう書いたかをコメントに残す**。何をしているかではなく、なぜその判断をしたかを書く
- 特にビジネスロジックの分岐には理由を必ず記載する

```php
// CI前: 物理削除（作業過程なので履歴不要）
// CI後: released履歴を残す（実際の部屋移動記録として必要）
if ($isCI) {
    $db->prepare("UPDATE room_assignments SET status = 'released' ...")->execute(...);
} else {
    $db->prepare("DELETE FROM room_assignments WHERE id = :id")->execute(...);
}
```

```javascript
// リサイズ完了後にonClickが発火して予約詳細に遷移してしまうため、
// フラグで100ms間クリックを抑制する
setTimeout(() => { isResizingRef.current = false; }, 100);
```

## 2. 日付処理（JavaScript）

- **`new Date('YYYY-MM-DD')` は使わない**。UTC解釈されてJST環境で日付がずれる
- 必ず `parseLocal()` を使う

```javascript
// ✕ NG: new Date('2026-04-10') → UTC 00:00 → JST 09:00（日付がずれる可能性）
const d = new Date('2026-04-10');

// ○ OK: ローカル時間で生成
function parseLocal(s) {
  const [y, m, d] = s.split('-').map(Number);
  return new Date(y, m - 1, d);
}
```

- **`toISOString()` も使わない**（UTCで出力されるため）。`fmt()` を使う

```javascript
// ✕ NG
d.toISOString().slice(0, 10);

// ○ OK
function fmt(d) {
  const y = d.getFullYear();
  const m = String(d.getMonth() + 1).padStart(2, '0');
  const day = String(d.getDate()).padStart(2, '0');
  return `${y}-${m}-${day}`;
}
```

## 3. PDO プリペアドステートメント（PHP）

- **同じパラメータ名を複数箇所で使えない**（`EMULATE_PREPARES = false` のため）

```php
// ✕ NG: :today を2箇所で使用 → SQLSTATE[HY093] エラー
$stmt = $db->prepare("
    SELECT * FROM room_assignments
    WHERE check_in_date <= :today AND check_out_date > :today
");

// ○ OK: 別名にする
$stmt = $db->prepare("
    SELECT * FROM room_assignments
    WHERE check_in_date <= :ci_date AND check_out_date > :co_date
");
$stmt->execute(['ci_date' => $today, 'co_date' => $today]);
```

---

## 4. SQLクエリ文字列（PHP）

- **ダブルクォートで囲む**。シングルクォートだとSQL内のリテラルのエスケープが必要になり、`Invalid parameter number` エラーの原因になる

```php
// ✕ NG: シングルクォート内でSQLリテラルをエスケープ
$stmt = $db->prepare('
    SELECT * FROM staff WHERE status = \'active\'
');

// ○ OK: ダブルクォートで囲めばSQLリテラルをそのまま書ける
$stmt = $db->prepare("
    SELECT * FROM staff WHERE status = 'active'
");
```

## 5. CI前 / CI後のアサイン操作

- **CI前のアサイン操作は `released` を残さない**。物理削除または上書きで良い（単なる作業過程）
- **CI後の部屋移動のみ `released` 履歴を残す**（お客様が実際に移動した記録）
- **CI後の操作はアサインボードからブロック**。予約詳細画面からのみ許可（`source: 'detail'` パラメータで制御）

---

## 6. 延泊 / 短縮の予約データ連動

- **延泊**: 予約CO日・nights・売上明細を連動更新する（延泊分の明細は金額0で自動生成）
- **短縮**: アサインの日程だけ変更。**予約データ・明細には触らない**
  - 理由: TL経由の予約CO日を勝手に変えるとOTAデータとの整合性が崩れる
  - 実際の途中退室は予約詳細のCO処理でキャンセルポリシーに従って明細を処理する

---

## 7. バー幅の計算（アサインボード）

- **JS計算（`span * cellWidth`）は使わない**。テーブルセルの実幅と計算値がずれる
- **CSS相対指定を使う**: `calc(泊数 * 100% - 2px)`。親セル幅を基準にするので常に正確

```javascript
// ✕ NG: JSで計算（セル幅の測定タイミングでずれる）
style={{ width: span * cellWidth }}

// ○ OK: CSSのcalcで親セル基準
style={{ width: `calc(${span} * 100% - 2px)` }}
```

---

## 8. ドラッグ操作と確認ダイアログ

- **全てのドラッグ操作に確認ダイアログを表示する**（誤操作防止）
  - 部屋移動: 「○○ を 302号室 → 403号室 に移動しますか？」
  - 新規アサイン: 「○○ を 302号室 にアサインしますか？」
  - 延泊/短縮: 「○○ のCO日を 04/12 → 04/14 に延泊しますか？」

---

## 9. リサイズとクリックイベントの干渉

- バーの右端ドラッグ（リサイズ）完了時に `onClick` も発火してしまう
- **`isResizingRef` フラグで `mouseup` 後100msの間 `onClick` を抑制する**

## 10. MySQL 8.4 互換性

- **`ADD COLUMN IF NOT EXISTS` は使えない**。INFORMATION_SCHEMA で存在チェックしてからALTER TABLEする

```sql
SET @col_exists = (SELECT COUNT(*) FROM INFORMATION_SCHEMA.COLUMNS
    WHERE TABLE_SCHEMA='pms_db' AND TABLE_NAME='staff' AND COLUMN_NAME='login_fail_count');
SET @sql = IF(@col_exists = 0,
    'ALTER TABLE staff ADD COLUMN login_fail_count INT NOT NULL DEFAULT 0', 'SELECT 1');
PREPARE stmt FROM @sql; EXECUTE stmt; DEALLOCATE PREPARE stmt;
```

## 11. usePolling と fetchData の更新タイミング

- `usePolling` は `[interval, enabled]` にのみ依存するため、`callback` が変わっても再実行されない
- **`dateRange` 等の変更で即座にデータを取得したい場合は、別途 `useEffect` で `fetchData()` を呼ぶ**

```javascript
// dateRange変更時に即座にフェッチ
useEffect(() => { fetchData(); }, [fetchData]);
// 定期ポーリング
usePolling(fetchData, 15000);
```

## 12. TLリンカーンへの書き込み禁止

- **TLリンカーンへの書き込みコードは絶対に書かない**（設計の根幹: 受動的PMS）
- TLからの受信データは原本として書き換えない:
  - `tl_last_name`, `tl_first_name` — 原本ゲスト名
  - `tl_checkin_date`, `tl_checkout_date` — 原本日程
  - `tl_room_type` — 原本部屋タイプ
- フロントで変更するのは `checkin_date` / `checkout_date` / `room_type` / `display_*` の方
- 予約詳細の「予約情報」セクションに、変更があった場合のみTL原本値を表示する

---

## 13. 物理DELETE禁止（原則）

- データの削除はステータス変更（論理削除）で行う
- **例外: CI前のアサイン操作**（作業過程であり履歴不要のため物理削除を許可）

## 14. ブラウザ標準ダイアログの使用禁止

- **`window.prompt()`, `window.confirm()`, `window.alert()` は使わない**
- 全てアプリ共通のダイアログ（`useConfirm` フック: `src/components/ConfirmDialog.jsx`）を使用する
- 理由: ブラウザ標準ダイアログはスタイルが統一できず、UXが悪い

### 共通ダイアログの3メソッド

```javascript
const { confirm: showConfirm, alert: showAlert, prompt: showPrompt } = useConfirm();
```

| メソッド | 用途 | 戻り値 |
|---------|------|--------|
| `confirm(title, message, options?)` | OK/キャンセル確認 | `true` or `false` |
| `alert(title, message)` | 通知（OKのみ） | `void` |
| `prompt(title, message, options?)` | 入力フィールド付き確認 | 入力値 or `null`（キャンセル時） |

### prompt の options

```javascript
const value = await showPrompt('タイトル', '説明文', {
  inputType: 'date',           // text, date, number 等
  defaultValue: '2026-04-16',  // 初期値
  min: '2026-04-16',           // 最小値（date/number用）
  max: '2026-04-19',           // 最大値
  placeholder: 'ここに入力',
  hint: '補足テキスト',
  confirmLabel: '実行する',     // デフォルト: 'OK'
  confirmColor: 'orange',      // blue, red, green, orange
});
if (value === null) return;    // キャンセル
```

### NG / OK 例

```javascript
// ✕ NG: ブラウザ標準
const value = prompt('日付を入力してください');
if (confirm('実行しますか？')) { ... }
alert('完了しました');

// ○ OK: 共通ダイアログ
if (await showConfirm('タイトル', '実行しますか？')) { ... }
showAlert('完了', '処理が完了しました');
const date = await showPrompt('日付入力', '分割日を選択', { inputType: 'date' });
```

## 16. 楽観ロック（同時更新の競合防止）

- 予約データを更新する操作（update / cancel / restore / merge / split / checkin / checkout）では、**リクエストに `updated_at` を含める**
- バックエンド側で DB の `updated_at` と比較し、不一致なら **409 Conflict** を返す
- 不一致 = 画面を開いた後に別スタッフが先に更新した → 「再読み込みしてください」エラー

### バックエンド

```php
// ReservationController / CheckinController に共通ヘルパーあり
$this->checkOptimisticLock($id, $body['updated_at'] ?? null);
```

- `updated_at` が送られていない場合はチェックをスキップ（後方互換）
- 新規APIを追加する場合は必ず楽観ロックチェックを入れること

### フロントエンド

```javascript
// 予約データの updated_at を各API呼び出しに渡す
await api.put(`/reservations/${id}`, { ...payload, updated_at: data.updated_at });
await api.post(`/reservations/${id}/checkin`, { updated_at: data.updated_at });
```

- APIクライアント（`api/client.js`）で 409 を共通ハンドリング済み
- 409 を受けたら画面再読み込みを促すエラーメッセージが表示される

## 19. ターミナルからのmysqlコマンドは `--default-character-set=utf8mb4` 必須

- アプリ（PHP/PDO）経由はDSNで `charset=utf8mb4` 指定済みなので問題ない
- **ターミナルから直接mysqlコマンドでINSERT/UPDATEする際は `--default-character-set=utf8mb4` を必ず付ける**
- 付けないとマルチバイト文字（日本語）が文字化けする
- 実際に文字化けが発生した事例あり（2026-04-14 card予約の入金行一括追加）

```bash
# ✕ NG: 文字化けする
mysql -u pms_user -p'PmsD3v2026!' pms_db -e "INSERT ..."

# ○ OK: UTF-8を明示
mysql -u pms_user -p'PmsD3v2026!' pms_db --default-character-set=utf8mb4 -e "INSERT ..."
```

## 20. コードの削除・残存判断（デッドコード扱い）

### 20.1 機械的判定だけで削除しない

`grep` で参照件数ゼロだったというだけでデッドコード判定しない。以下を必ず確認:

1. **本来使うべき場所で使い忘れていないか**（例: `getCountryName()` は導入済みだがGuestDetailPageで `country_code` を生コード表示していた → 使い忘れ）
2. **FK制約など実装上の不具合で一時的に呼出しを外しただけではないか**（例: `cleanupOrphanGuest` はFKエラー対応で呼出し停止していたが、本来の意図は取消ゲスト掃除）
3. **将来実装のための設計パーツではないか**（例: `.ab__bar--released` は released履歴を薄く表示するためのスタイル、今は非表示処理だが将来復活の可能性あり）
4. **他ファイルとペアになっているか**（例: `.ab__bar-link` は RoomIndicator の `has_link` 🔗アイコンと対をなす想定）

### 20.2 削除前の承認手順

1. 削除候補をリスト化する（ファイル:行番号 + 何が・なぜデッドに見えるか）
2. ユーザーに承認を取る
3. 承認後に削除実行

一括で「チェックして削除」と指示された場合でも、削除対象の一覧をまず提示すること。

### 20.3 意図的に残すコード（dead に見えるが必要）のコメント

JSX/呼出し側で使用されていないが意図的に残すコードには、必ずWHYコメントを書く:

- **なぜ削除しないのか**（将来実装用？他モジュールとペア？拡張を想定した引数？）
- **何を壊す可能性があるか**（削除すると再定義が必要、等）

```css
/* ab__bar-link は現状JSX未使用だが、RoomIndicatorの has_link 🔗アイコンと
   対をなすアサインボード側のバッジとして将来実装する想定で残す。
   削除すると個別バー上に連結識別バッジを戻す際に再定義が必要になるため保持。 */
.ab__bar-vip, .ab__bar-link { ... }
```

```php
/**
 * $reservationNo を引数に残している理由:
 *   将来、reservation_no + DataID の組み合わせで重複検知する可能性がある。
 *   シグネチャを維持して拡張時に呼出側の変更を最小化する。
 */
private function isDuplicate(string $dataId, string $reservationNo): bool
```

### 20.4 実際に起きた誤削除事例（2026-04-24）

| 対象 | 誤判定の原因 | 本来の意図 |
|------|-------------|-----------|
| `cleanupOrphanGuest()` | 呼出しが削除されていたため未使用に見えた | 取消のみのゲストを顧客一覧から消すため（実装はされていたが、FK制約エラーで一時的に呼出しを外していた） |
| `getCountryName()` | import ゼロ件だった | GuestDetailの国籍欄で「日本（JP）」と表示すべきだった（使い忘れ） |
| `.ab__bar--released` | JSXで `status='released'` が非表示処理になっていた | 将来「移動履歴をグレーアウト表示」に戻す設計余地として残していた |
| `.ab__bar-link` | JSX未使用 | RoomIndicatorの `has_link` 🔗アイコンと対をなす将来用バッジ |

## 21. UIデザイン言語「Patina」（2026-07刷新）

詳細は `docs/UI刷新_Patina_2026-07.md` を参照。コーディング上の要点:

- **`--accent-blue` の値はブラス（真鍮）色 #8A7440**。変数名は旧デザインからの互換のため維持している。青に戻さない・新規コードでも主要アクション色はこの変数を使う
- **新規UIでハードコード色を書かない**。variables.css のトークン（--bg-hover / --bg-inset / --border-strong / --accent-blue-soft / --accent-blue-ring 等）を使う
- フォントサイズの下限: 本文・ラベルは12px、バッジ・タグ類のみ11px許容。9px以下は禁止
- 入力欄は `border: 1px solid var(--border-strong)`、focus時 `border-color: var(--accent-blue)` + `box-shadow: 0 0 0 3px var(--accent-blue-ring)` で統一
- ステータスバッジはピル型＋先頭ドット（styles/status-badge.css）。OTAバッジはベタ塗り維持（識別性優先）

## 22. CRA(css-loader)ではCSSから public/ 配下の画像を参照できない

- CSS内の `url(/coral-logo.svg)` や `content: url(/...)` は css-loader がモジュール解決を試みて **ビルドエラー** になる（実際に発生: 2026-07-02 ダークテーマのロゴ差し替え）
- public/ の画像は JSX の `<img src="/...">` でのみ参照する
- ダークテーマ等でCSSだけで画像の見た目を変えたい場合は `filter`（invert+hue-rotate等）を使う（Sidebar.css / Header.css 参照）

## 23. Cloudflareエッジキャッシュと public/ 配下の静的アセット

- 本番はCloudflareプロキシ経由のため、**SVG・ICO・PNG等の静的アセットはエッジで約4時間キャッシュされる**
- `frontend/public/` 直下のファイル（ロゴSVG・favicon等）はCRAのビルドハッシュが付かないため、**差し替えてデプロイしてもCloudflareが旧版を返し続ける**（ブラウザの強制再読み込みでは回避不能。実際に発生: 2026-07-07 ロゴ白抜き変更）
- `static/js|css` 配下はファイル名にハッシュが付くので問題ない
- **対処（自動化済み 2026-07-07）**: deploy.sh がデプロイ後にCloudflare APIでルート直下ファイルのURLを自動パージする
  - APIトークン: `~/.config/pms/cloudflare_token`（キャッシュパージ＋ゾーン読取のみの最小権限。リポジトリに含めない）
  - トークンファイルが無い環境ではパージをスキップして警告表示（その場合は手動パージ or 最大4時間待ち）
  - JSXから参照するpublic/直下アセットには念のため `?v=N` を付けている（Sidebar.jsx参照）。差し替え時は番号を上げる

## 24. ルート定義の順序（固定パスは `:id` より前に置く）

- ルーターは登録順にマッチするため、`/documents/:id` を先に登録すると
  `/documents/sales-receipt` が「id='sales-receipt'」として解釈されてしまう
- **同じ階層に固定パスと `:id` がある場合、固定パスを先に登録する**（2026-07 物販の即売領収書で対応）

## 25. 物販: product_sales と reservation_charges は対で扱う

- 部屋付けの物販は `product_sales` 1行につき `reservation_charges`（charge_type='goods'）1行を生成し、
  `product_sales.charge_id` で紐付けている
- **片方だけを更新・取消してはならない**（請求額と売上集計が食い違う）。取消は必ず
  `ProductSaleController::cancel()` を通し、両方を cancelled にして予約の amount を再計算する
- そのため予約詳細の明細テーブルでは goods 行の編集・削除ボタンを出していない（鍵アイコン表示）。
  明細の手動追加セレクトからも goods を除外している
- 即売（reservation_id = NULL）は `reservation_charges` に行を作らない。
  領収書は `document_items.sale_id` で紐付ける

## 26. 領収書の税率区分記載（インボイス）

- `ReceiptPdfService::summarizeByTaxRate()` は **軽減税率の明細が1件も無ければ標準税率の1区分だけを返す**
- 標準税率の対象額は「合計 − 宿泊税 − 軽減税率分」の逆算で求める。
  宿泊料は明細に消費税額を持たない運用のため、この逆算をやめると既存領収書の表示額が変わる
- **物販導入前に発行した領収書のPDFが同じ見た目のままであること**を、変更時は必ず確認する

## 27. 物販の売上は経路が2つある（二重計上に注意）

| 販売形態 | reservation_charges | product_sales |
|---------|--------------------|---------------|
| 部屋付け | goods行あり（予約の請求額・宿泊領収書に反映） | あり |
| 即売     | なし                                          | あり |

- 収入実績・予測表（`IncomeReportService`）は **両方を「売上金額」に含める**（2026-07-22 決定）
  - 部屋付け: `charge_type IN ('room','addon','discount','goods')` に goods を追加
  - 即売: `product_sales`（`reservation_id IS NULL`）を日別に集計して加算
  - **即売分を加算する際は必ず `reservation_id IS NULL` で絞ること**。部屋付けは goods 明細として
    既に計上済みであり、絞り忘れると二重計上になる
  - 内数はPDFの「物販」列に出している（`/reports/products` の合計と一致する）
- `/reports/products` は物販の内訳を見るためのもの。収入実績表の売上金額に足し合わせないこと
