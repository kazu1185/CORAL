## 実機再現に成功したバグ（最優先）

### #1【高】統合解除でCO日当日の入金・明細が親に取り残される

- 場所: `ReservationController::restoreChildFromSource()` — `date >= :ci AND date < :co` で明細を子に戻す
- 統合時は **reservation_id 単位で全明細**（日付不問）を親へ移すのに、解除時は **日付範囲**で戻すため非対称。
  CO日当日の入金（チェックアウト精算の典型）・CO日朝食addon・滞在期間外日付の前受金が親に残る。
- **実機再現**: 予約B(6/22-24, 24,000円)にCO日6/24付の入金24,000円を記録→Aと統合→解除。
  結果: **B = 未収24,000円 / A = 過入金4,000円** の会計ずれが発生（領収書・精算画面に直撃）
- 修正方針案: 統合時に明細へ元予約IDを記録して解除時はそれで戻す（カラム追加）、
  または解除時の戻し条件を「入金/返金行は日付不問で元source範囲に従う」等に再設計

### #2【高】CO日当日で分割すると0泊・0円のconfirmed予約が生成される

- 場所: バックエンド `split()` のバリデーション `splitDate > checkout_date`（CO日ちょうどを許容）
  + フロント分割ダイアログ `max: d.checkout_date`（ReservationDetailPage）
- **実機再現**: 統合予約(6/25-27)をCO日6/27で分割 → **ID1365: CI=CO=6/27, nights=0, amount=0, confirmed** が誕生。
  予約一覧・在庫・ダッシュボードに0泊のゴミ予約が混入する
- 修正方針案: バックエンドを `splitDate >= checkout_date` で422に、フロントmaxを `addDays(co, -1)` に

## コードで確認した高リスク（未再現だが確実）

### #3【高】CI済み予約の分割でアサインと履歴が全消去される

- `split()` はstatus `checked_in` を許可した上で、`DELETE FROM room_assignments WHERE reservation_id = :id`
  を **statusフィルタなし** で実行。滞在中の部屋割りも、CI後の `released` 移動履歴も物理削除される
- 規約#13（物理削除はCI前の作業過程のみ）違反 + 在室中ゲストの部屋が分からなくなる
- 修正方針案: CI済みの分割を禁止する、または active のみ削除+履歴保全+前半へのアサイン引継ぎ

### #4【高】複数室グループの子予約を統合できてしまう

- `merge()` のガードは `status='confirmed'` のみ。`parent_reservation_id` を持つグループ子も統合可能
- グループ子は `reservation_no=NULL`（TlImportService）のため、sourceが `reservation_no=NULL` で記録され
  TL変更通知の照合（channel+resno）が永遠にヒットしない。親の room_count と子数の整合も崩壊
- 修正方針案: merge() で `parent_reservation_id IS NOT NULL` / `room_count > 1` を422で弾く

### #5【高】複数のmerge_alertがあると2件目以降が静かに失われる

- `resolveMergeAlert()` は指定した1件のアラートの変更内容だけを復元予約に反映する。
  フロント `MergeAlertResolveDialog` も `alerts[0].event_id` のみ送信（「全解除なので1回で済む」とのコメントは誤り）
- 全解除後は `is_merged_parent=false` になるため、未処理アラートは `pending_merge_alerts` の
  算出対象外となり画面から消える → **別OTAのキャンセル通知が予約に反映されないまま誰も気付けない**
  （OTA側ではキャンセル済み・PMSではconfirmed → ゴースト予約）
- 修正方針案: resolveMergeAlert で同一予約の未処理アラートを全件ループ反映する

### #6【高】グループ一括CI/COに楽観ロックがない

- `groupCheckin()` / `groupCheckout()` は `checkOptimisticLock` を呼ばない（単体CI/COは呼ぶ）。
  フロント（GroupReservationPage / GroupReservationCard）も updated_at を送っていない
- 複数端末運用（フロント6名）で同時操作時に競合検出不能

### #7【高】visit_count（来館回数）の多重加算

- グループCOを**部分選択で複数回**に分けると、呼び出しごとに同一ゲストへ +1（CheckinController 411-420）
- さらに子予約を**単体checkout()**でCOすると子1室ごとに +1（同 172-181）→ 3室グループで+3
- 「1滞在=1回」の意図（コードコメントに明記）が複数経路で破られる。リピーター判定・来館表示が汚染される
- 修正方針案: visit_count加算を「同一guest_id+同一滞在期間で当日加算済みならスキップ」にする等
