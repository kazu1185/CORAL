# 予約統合機能 設計メモ

## 背景・要件

OTAによっては連泊を1泊ずつ別予約で通知するケースがある。
例: 4泊の滞在が4件の1泊予約としてTLから届く。

これらを**1つの予約（reservationsの1レコード）として統合**し、
CI/CO・アサイン・清掃・領収書の全ロジックが通常の連泊予約と同様に動作する必要がある。

## 設計思想（実装済み）

### reservations = 1レコード = 1滞在

通常予約も統合予約も同じテーブル・同じカラムで扱う。

| ケース | reservation_no | channel | reservation_sources |
|--------|---------------|---------|-------------------|
| 通常予約（TLから1件） | OTA番号そのまま | OTAチャネル名 | なし |
| 統合予約（複数OTAを1滞在に） | NULL | NULL | 全OTA予約番号を保持 |

- PMS内部の予約番号は `reservations.id`（主キー）
- 統合後は特定OTAに属さない → `reservation_no = NULL`, `channel = NULL`
- 元のOTA予約番号は**全て**（親自身の分も含めて）`reservation_sources` に保持
- チャネル混在OK（同一ゲストが異なるOTAから1泊ずつ予約するケースがあるため）

### reservation_sources テーブル

```sql
reservation_sources (
    id INT AUTO_INCREMENT PRIMARY KEY,
    reservation_id INT NOT NULL,            -- 統合先の予約ID
    original_reservation_id INT NULL,       -- 元の予約レコードID
    reservation_no VARCHAR(50) NOT NULL,    -- 元のOTA予約番号
    channel VARCHAR(30) NOT NULL,
    checkin_date DATE NOT NULL,
    checkout_date DATE NOT NULL,
    amount INT NOT NULL,
    nights INT NOT NULL,
    status ENUM('active','cancelled') NOT NULL DEFAULT 'active',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (reservation_id) REFERENCES reservations(id),
    INDEX idx_reservation_no (reservation_no, channel),
    INDEX idx_reservation_id (reservation_id),
    UNIQUE KEY uk_channel_resno (channel, reservation_no)
)
```

## 統合の処理フロー（実装済み）

1. スタッフが予約一覧で複数予約をチェックボックスで選択し「統合」ボタンを押す
2. 選択された予約の日程が連続しているか検証（チャネル混在OK）
3. CI日でソートし先頭の予約を「親」レコードとして使用
4. **全予約**（親自身も含む）の情報を `reservation_sources` に記録
5. 子予約の日別明細（reservation_charges）を親予約に移行
6. 親予約の `reservation_no = NULL`, `channel = NULL` に変更（OTA非依存化）
7. 親予約の `checkin_date`, `checkout_date`, `nights`, `amount` を全体に拡張
8. 子予約を `status = 'merged'`, `guest_id = NULL` に変更
9. `reservation_events` に統合イベントを記録
10. `staff_activity_logs` に操作ログを記録

## TL通知の処理フロー（実装済み）

TL通知が来た時の検索順序:
1. `reservations.reservation_no` で検索（通常予約）
2. 見つからなければ `reservation_sources.reservation_no` で検索（統合予約）
3. reservation_sources で見つかった場合 → 統合予約への影響:
   - **取消通知** → `merge_alert` イベント記録（自動処理しない）
   - **日程変更**（CI/COが変わった）→ `merge_alert` イベント記録（自動処理しない）
   - **金額変更のみ**（日程不変）→ charges を自動更新 + source.amount も更新
4. スタッフがダッシュボードのアラートを見て判断・対応

**原則: 統合予約への日程変更・取消は自動処理せず、アラートでスタッフに判断を委ねる**

## 分割機能（実装済み）

- `POST /api/v1/reservations/:id/split` — 統合予約を指定日で2分割
- **unmerge（元に戻す）ではない** — 任意の日付で前半/後半に分割する操作
- 用途: 統合済み予約の中日がキャンセルされた場合等
- 後半用に新規予約レコードを作成（`reservation_no` に `-S` suffix）
- charges と sources を split_date で振り分け

## UI表示

### 予約一覧
- `channel = NULL` → 紫色の「統合」バッジ
- `reservation_no = NULL` → `#ID` 形式で表示（例: `#1627`）
- merged 予約は一覧から除外

### 予約詳細
- トップバー: 「統合」バッジ + `#ID`
- 右カラム: 「統合元予約」カード（全OTA予約番号一覧 + 分割ボタン）
- 予約情報セクション: PMS予約番号（`#ID`）を表示。通常予約のみOTA予約番号/チャネルも表示

### ステータス件数
- merged はカウントから除外（`WHERE status != 'merged'`）

## 楽観ロック（実装済み）

- merge / split / update / cancel / restore / checkin / checkout の全操作で `updated_at` チェック
- 他スタッフが先に更新していた場合は 409 Conflict を返す
- フロントは画面を開いた時点の `updated_at` を各APIリクエストに含める

## 関連テーブル
- `guest_links`: 複数室予約の緩い連結。統合とは別概念。TlImportServiceが複数室予約で使用
- `reservation_events`: 統合・分割・merge_alert 操作を記録
- `reservation_charges`: 日別明細を統合先に移行

## 注意事項
- 通常の1予約（TLから4泊で届いたもの）は reservation_sources を持たない
- 統合元の tl_last_name 等の原本データは merged 化された子予約レコードに残る
- amount は reservation_charges の合算で再計算
- reservations.channel は NULL 許可（統合予約用）
