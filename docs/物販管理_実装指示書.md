# 物販管理機能 実装指示書（2026-07-22作成）

フロントデスクでの物販（土産・飲料・アメニティ等）を管理する機能の実装指示書。
設計はFable 5セッションで確定済み。**この指示書と規約に従って実装すること。**

## 必読（実装開始前）

1. `CLAUDE.md` — 絶対ルール（承認なしのコード変更禁止・物理DELETE禁止・TL書き込み禁止）
2. `docs/コーディング規約・注意事項.md` — **全項目**。特に #2(日付) #3(PDO別名) #4(SQLダブルクォート) #10(MySQL8.4マイグレーション) #14(useConfirmダイアログ) #15(共通パーツ必須) #16(楽観ロック) #21(Patinaトークン) #23(Cloudflareキャッシュ)

## 確定済み要件

| 項目 | 決定内容 |
|------|---------|
| 販売形態 | **部屋付け**（宿泊予約の明細に追加、CO精算）＋**即売**（現金/カード等の即時決済）の両方 |
| 在庫管理 | **なし**（将来拡張の余地だけ残す。在庫列・入荷機能は作らない） |
| 税率 | **8%（軽減）と10%混在**。商品ごとに保持。領収書は税率区分記載（インボイス対応） |
| UI | サイドバーに「物販」専用ページ（レジ風）＋予約詳細からも部屋付け追加可 |
| 即売領収書 | **v1に含める**（documents.reservation_id は既にNULL許容なので小改修で可能） |

## 1. DBマイグレーション `backend/migrations/022_product_sales.sql`

規約#10に従い、列追加・enum変更は INFORMATION_SCHEMA で存在チェックしてから実行する形式で書くこと。

### 1-1. products（商品マスタ・新規）

```sql
CREATE TABLE IF NOT EXISTS products (
  id INT AUTO_INCREMENT PRIMARY KEY,
  product_name VARCHAR(100) NOT NULL,
  category VARCHAR(30) NOT NULL DEFAULT 'その他',   -- レジ画面のタブ分け用（飲料/食品/雑貨 等の自由入力）
  price INT NOT NULL,                                -- 税込価格（円）
  tax_rate TINYINT NOT NULL DEFAULT 10,              -- 8 or 10。ハードコード禁止規約のためDBで保持
  sort_order INT NOT NULL DEFAULT 0,
  is_active TINYINT(1) NOT NULL DEFAULT 1,           -- 論理削除
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 1-2. product_sales（販売記録・新規）— 物販の記録の本体

```sql
CREATE TABLE IF NOT EXISTS product_sales (
  id INT AUTO_INCREMENT PRIMARY KEY,
  sale_date DATE NOT NULL,
  reservation_id INT NULL,           -- NULL=即売 / あり=部屋付け
  charge_id INT NULL,                -- 部屋付け時に生成した reservation_charges 行
  product_id INT NOT NULL,
  product_name VARCHAR(100) NOT NULL, -- スナップショット（マスタ変更後も過去売上を不変に保つ）
  unit_price INT NOT NULL,            -- スナップショット（税込）
  tax_rate TINYINT NOT NULL,          -- スナップショット
  quantity INT NOT NULL DEFAULT 1,
  amount INT NOT NULL,                -- unit_price * quantity（税込）
  tax_amount INT NOT NULL,            -- 内消費税額（端数は円未満切り捨てで統一）
  payment_method_id INT NULL,         -- 即売時のみ。部屋付けはNULL（CO精算時に決まる）
  staff_id INT NOT NULL,
  status ENUM('active','cancelled') NOT NULL DEFAULT 'active',  -- 物理DELETE禁止
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  KEY idx_sale_date (sale_date),
  KEY idx_reservation (reservation_id),
  KEY idx_product (product_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

内税額の計算: `tax_amount = amount - floor(amount / (1 + rate/100))`（税込価格から逆算、円未満切り捨て）。フロント・バック両方この式で統一。

### 1-3. 既存テーブル改修

- `reservation_charges.charge_type` の ENUM に **'goods'** を追加（`012_add_discount_charge_type.sql` と同じ MODIFY COLUMN 方式。既存値を全て列挙し 'goods' を足す）
- `reservation_charges` に `tax_rate TINYINT NULL` を追加（既存行はNULL=10%扱い。goods行のみ8/10を設定）
- `document_items` に `tax_rate TINYINT NULL` と `sale_id INT NULL` を追加（領収書の税率区分記載＋即売明細の参照用）
- `permissions` / `role_permissions` に以下を追加。**既存の permissions テーブルの実データを必ず確認し、命名・role割当の流儀に合わせること**：
  - `product_sales.view`（物販ページ閲覧）・`product_sales.create`（販売・取消）→ フロント系ロールに付与
  - `master.products`（商品マスタ管理）→ 既存の master.plans と同じロールに付与

### 1-4. ローカルDBへの適用

規約#19: ターミナルからは `--default-character-set=utf8mb4` 必須。

## 2. バックエンドAPI

### 2-1. 商品マスタ（MasterController に追加）

既存の plans 系と完全に同じパターンで実装（`updateMaster()` ヘルパー使用）:

```
GET  /api/v1/master/products          permission: reservation.view（一覧は販売画面でも使うため）
POST /api/v1/master/products          permission: master.products
PUT  /api/v1/master/products/:id      permission: master.products
POST /api/v1/master/products/reorder  permission: master.products
```

### 2-2. 販売（新規 ProductSaleController）

```
POST /api/v1/product-sales            permission: product_sales.create
GET  /api/v1/product-sales?date=YYYY-MM-DD&range=…  permission: product_sales.view
PUT  /api/v1/product-sales/:id/cancel permission: product_sales.create
```

**POST（販売確定）** — body: `{ items: [{product_id, quantity}], reservation_id?, payment_method_id?, updated_at? }`
- 商品情報は**サーバー側でマスタから読み直してスナップショット**する（フロントから単価を受け取らない。改ざん防止）
- 即売: `payment_method_id` 必須、`reservation_id` NULL
- 部屋付け: `reservation_id` 必須。対象予約が **CI済み（in_house）であること**をチェック。`reservation_charges` に charge_type='goods' の行を同時生成（description=「商品名 ×N」、date=販売日、tax_rate設定）し、`charge_id` に紐付け。**規約#16の楽観ロックを適用**（`checkOptimisticLock`）。amount再計算が必要なら既存の `recalcAmountFromCharges()` の流儀に従う
- 複数商品の一括販売は商品ごとに product_sales 行を作る（1トランザクション）

**cancel（取消）** — 論理削除。部屋付けは紐づく reservation_charges 行も status='cancelled' に連動（楽観ロック適用）。即売はそのまま cancelled に。**物理DELETEは書かない**

### 2-3. 即売領収書（DocumentController / DocumentService 拡張）

```
POST /api/v1/documents/sales-receipt  permission: receipt.issue
```
- body: `{ sale_ids: [...], addressee, payment_method_id }`
- documents 行: type='receipt'、**reservation_id=NULL**、addressee=入力値
- document_items: sale_id で紐付け、tax_rate をスナップショット
- 番号採番・再発行・取消は既存の受領書フローに準拠

### 2-4. 領収書の税率区分記載（インボイス対応）

- DocumentService: 明細作成時に charge/sale から tax_rate を引き継ぐ（NULLは10%扱い）
- 領収書PDF（TCPDF）: 合計欄の下に「10%対象 ¥X（内消費税 ¥Y）」「8%対象 ¥Z（内消費税 ¥W）」を出力。8%対象が0件なら8%行は出さない（既存領収書の見た目を変えない）
- 規約#17: SVGは使わない。既存レイアウトへの追記に留める
- 規約#18: goods行のdescriptionはそのまま使用（プラン名差し替えの対象外）

### 2-5. レポート（ReportController 拡張）

- `GET /api/v1/reports/products?from=&to=` — 商品別（数量・金額）、税率別、即売/部屋付け別、支払方法別（即売のみ）の集計
- 既存 daily / monthly に**即売分の売上を追加**（部屋付け分は reservation_charges 経由で既存集計に乗っているため二重計上しないこと。集計にどちらの経路で含まれるかをコメントで明記）

## 3. フロントエンド

### 3-1. 物販ページ（新規 `pages/ProductSalesPage.jsx`）

サイドバー「清掃管理」の下に「物販」を追加（permission: `product_sales.view`）。

- **商品グリッド**: カテゴリタブ＋税込価格表示の大きめタッチボタン（清掃タブレットでも押せるサイズ、最低44px）。activeな商品のみ、sort_order順
- **カート**: 商品タップで追加、数量±、小計と内税（8%/10%別）表示
- **確定**: 「即売」→ 支払方法選択（payment_methodsマスタ、activeのみ）／「部屋付け」→ **在室中（CI済み）の予約一覧から選択**（部屋番号・ゲスト名で検索）。いずれも useConfirm の確認ダイアログ必須（規約#14）
- **当日の販売履歴**: ページ下部に一覧（時刻・商品・数量・金額・即売/部屋番号・スタッフ）。取消ボタン→確認ダイアログ→cancel API。取消済みは打ち消し線表示
- **即売領収書**: 履歴行から「領収書」→ 宛名入力（useConfirmのprompt）→ 発行→PDF表示（既存領収書と同じ動線）

### 3-2. 予約詳細への組み込み

- 明細セクションに「物販追加」ボタン（CI済みの予約のみ表示）
- 商品グリッド＋カートを**モーダルで再利用**（共通コンポーネント `components/ProductPicker.jsx` に切り出すこと。ページとモーダルで同一実装を使う。規約#15の精神）
- 追加後は明細一覧に goods 行が現れる（既存の明細表示に乗る）

### 3-3. 商品マスタ画面（`pages/settings/ProductSettings.jsx`）

- **必ず `useMasterCrud` + `useDragReorder` を使う**（規約#15。fetch/保存/トグル/D&Dのコピペ禁止）
- 項目: 商品名・カテゴリ・税込価格・税率（10%/8%のセレクト）・有効/無効・D&D並び替え
- 設定メニューへの追加は既存の設定ページ一覧の実装に合わせる（permission: `master.products`）

### 3-4. 売上レポートへの追加

- 物販セクション: 商品別売上（数量・金額）、即売/部屋付け内訳、支払方法別（即売）
- 期間指定は既存レポートのUIに合わせる

### 3-5. 共通事項

- 日付は `utils/date.js` の `parseLocal/fmt/todayStr` のみ使用（規約#2）
- 色は variables.css のトークンのみ（規約#21。`--accent-blue`=ブラス色に注意）。フォントサイズ下限12px
- ラベル定数が複数ファイルで必要になったら `utils/constants.js` へ（税率表示「軽減8%」等）
- ポーリングは物販ページでは不要（自画面操作が主。履歴はアクション後に再取得）

## 4. 実装順序と動作確認

1. **マイグレーション＋商品マスタAPI＋マスタ画面** → 商品を数件登録できること
2. **販売API＋物販ページ（即売）** → 即売→履歴表示→取消の一連
3. **部屋付け** → CI済み予約に追加→予約詳細の明細に反映→CO精算・領収書に乗る→取消連動
4. **即売領収書＋税率区分記載** → 8%/10%混在カートで発行し、区分記載を確認。既存の宿泊領収書（10%のみ）の見た目が変わらないことも確認
5. **レポート** → 二重計上がないこと（部屋付け商品が既存売上とproductsレポートの両方で辻褄が合うこと）

各フェーズ完了時にコミット。テストデータでの検証は開発DBで行う（本番DBには触らない）。
