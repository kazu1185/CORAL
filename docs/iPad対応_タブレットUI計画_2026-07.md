## 1. 基本方針: PCとiPadの棲み分け

2026-07-24 ユーザー決定。全画面をタッチ対応させるのではなく、**業務でデバイスを分担する**。

| デバイス | 担当業務 | UI方針 |
|---------|---------|--------|
| **iPad（13インチ）** | **対面カウンター業務**: チェックイン、チェックアウト（精算・領収書含む） | 新設「**フロントモード**」— タッチ専用に新規設計。大型ターゲット・PWA |
| **PC** | 管理業務: アサイン、予約取り込み・予約管理、締め作業・レポート、在庫、設定 | **現状維持。タッチ改修はしない** |

### この方針で「やらない」と決めたこと（旧計画から削除）

初版計画（全画面タッチ対応）の以下の大工事は**不要になった**。将来必要になった場合に備え、調査結果は付録Aに残す。

- アサインボードの Pointer Events 化（D&D・リサイズ・右クリックメニューのタッチ対応）→ アサインはPC業務
- `useDragReorder`（マスタ並び替え4画面）のタッチ対応 → 設定はPC業務
- 部屋インジケーター編集モードのタッチ対応 → レイアウト編集はPC業務
- サイドバー折りたたみトグルの改修 → フロントモードにはサイドバー自体が無い
- 既存全画面の44ptタップターゲット化・タブレットブレークポイント → フロントモードのみ新規で適用

※ iPadから既存の管理画面を開くこと自体は禁止しない（閲覧は可能）。ただし最適化対象外であり、アサインボードのD&Dがタッチで動かないのは**仕様**とする。
※ 将来の清掃スタッフ画面（Phase 13、現在プレースホルダ）は、フロントモードで作るタッチ基盤（レイアウト・大型部品）を土台に最初からタッチ前提で設計する。

## 2. ターゲット環境

| 項目 | 値 |
|------|-----|
| デバイス | iPad Pro 13インチ（M4）。論理解像度 横1376×1032pt / 縦1032×1376pt |
| ブラウザ | Safari + **ホーム画面追加（PWA standalone）を正式な起動方法とする** |
| 操作 | タッチ（指）のみで全カウンター業務が完結すること |
| 想定シーン | フロントカウンターに常設。スタッフが対面接客しながら操作。縦横どちらでも成立させる（横を基本とする） |

### 「超一流」の定義（フロントモードに適用）

1. 対面業務がタッチだけで、**迷いなく最短手数で**完結する（チェックインは一覧から2〜3タップ）
2. Apple HIG 準拠の44ptタップターゲット。押し間違いが構造的に起きない
3. ネイティブアプリと見紛う挙動: 入力時の勝手なズームなし、下端見切れなし、standalone起動でフルスクリーン、滑らかな画面遷移
4. **接客中に見られても美しい画面**（Patinaデザイン言語: アイボリー#F4F2ED・墨色#26221B・ブラス#8A7440。variables.cssトークンのみ使用、ハードコード色禁止 = 規約 #21）
5. 共用端末前提の速いログイン（大型PINパッド）

## 3. フロントモードの設計

### 3.1 構成: 専用ルート + 専用レイアウト

- ルート: `/front` 配下に新設（例: `/front` = 本日ボード、`/front/checkin/:id`、`/front/checkout/:id`）
- **`FrontLayout` を新設**（既存 `Layout`/`Sidebar` は使わない）:
  - サイドバーなし。上部に大型タブナビ（本日のチェックイン / 本日のチェックアウト / ＋候補機能のタブ）
  - ヘッダーに現在日時・ログイン中スタッフ・「管理画面へ」リンク（既存画面への脱出口）
  - `height: 100dvh` + セーフエリア対応をレイアウトレベルで担保
- 既存画面側には、サイドバーに「フロントモード」への入口を1つ追加（iPadでブックマーク/PWA起動URLを `/front` にすれば通常は行き来不要）
- モード記憶: 端末の localStorage に「この端末はフロント端末」フラグを保存し、ログイン後の初期遷移先を `/front` にする（清掃ボードのデバイストークン方式と同じ発想。ただし認証自体は既存のスタッフ名+PINセッションを使う）

### 3.2 本日ボード（フロントモードのホーム）

- データ源: 既存 `DashboardController::index`（本日のCI/CO一覧）。不足があればAPI拡張ではなく**既存レスポンスの範囲で開始**し、必要になった項目のみ追加検討
- 2タブ（または左右2ペイン）: 「本日のチェックイン」「本日のチェックアウト」
- 1予約 = **大型カード**（高さ72pt以上）: ゲスト名（原本tl_*は書き換えない= 規約 #12）、部屋番号、人数、泊数、OTAバッジ（styles/ota-badge.css）、ステータスバッジ（styles/status-badge.css）、未アサイン・要注意アラート
- 5〜10秒ポーリングで自動更新（既存 `usePolling` + 規約 #11 の fetchData 併用パターン）
- 検索: ゲスト名のインクリメンタル検索（ウォークイン客の当日予約照会用）

### 3.3 チェックインフロー（Phase 2）

既存API `POST /reservations/:id/checkin`（CheckinController::checkin）＋グループ用 `groupCheckin` を使用。**バックエンドは原則変更しない**。楽観ロック（規約 #16: updated_at送信）必須。

画面フロー（1画面1目的のステップ型。戻るボタン常設）:

1. 本日ボードでカードをタップ → **チェックイン確認画面**
   - 予約内容の大きな表示（ゲスト名・日程・部屋・プラン・人数・支払状況）
   - 部屋未アサインの場合: **CI不可の明確な表示**＋「PCでアサインしてください」ガイダンス（アサインはPC業務。iPadからはアサインさせない）
2. ゲスト情報の確認・不足入力（連絡先等。既存ゲストAPI使用）
3. CI実行 → 成功演出（チェックマーク）→ 本日ボードへ戻る
- グループ予約はグループCI（groupCheckin）の一括実行UIを用意
- 確認・エラーはすべて `useConfirm`（規約 #14）。409（楽観ロック）時は再読み込み誘導

### 3.4 チェックアウト・精算フロー（Phase 3）

既存API `POST /reservations/:id/checkout` ＋ `groupCheckout`、明細は reservation_charges、領収書は DocumentController / ReceiptPdfService を使用。

1. 本日ボードでカードをタップ → **精算画面**
   - 明細一覧（大きな文字・行高）: 宿泊・追加・物販（goods行は編集不可 = 規約 #25 どおり）・宿泊税・入金済み額・**残額を最大サイズで表示**
2. 追加入金の登録（支払方法ボタン: 現金/カード/その他マスタ駆動）
3. 領収書発行: 宛名入力（大型キーボード対応）→ PDF発行 → **iPadからのAirPrint印刷を実機確認**（PDFを新規タブで開く現行方式がiPad Safariでどう振る舞うか要検証。standaloneモードでのPDF表示・印刷が最重要検証項目）
4. CO実行 → 本日ボードへ戻る
- 途中退室・キャンセルポリシー絡みの複雑な精算は**PCの予約詳細に誘導**する（フロントモードは正常系に集中。規約 #6 の短縮処理はiPadに持ち込まない）

### 3.5 候補機能（Phase 4。実装前にユーザーと採否を相談）

| 候補 | 内容 | 判断材料 |
|------|------|---------|
| 物販即売POS | 既存 ProductSalesPage をフロントモードに組み込み、タッチ最適化（数量ボタン24px→44pt等） | 対面販売はカウンター業務なのでiPad親和性が高い。既存画面の改修で済む |
| 部屋状況ビュー | RoomIndicator の**閲覧専用版**（編集モード非搭載） | 接客中に「今どの部屋が空いてるか」を見る用途。閲覧だけならタッチ改修は軽微 |
| パスポート撮影 | CIフロー中に `<input type="file" accept="image/*" capture="environment">` でiPadカメラ起動→ 既存 storage/passports へアップロード | 外国人ゲストのCIが1台で完結する。バックエンドのアップロードAPI有無を要確認 |
| レジカード署名 | 将来候補としてメモのみ（Canvas署名）。本計画では実装しない | — |

## 4. 実装フェーズ詳細

### Phase 0: 土台（全アプリ共通。見積: 小）

棲み分け型でもここは全画面に効くので最初にやる。

1. **index.html**: viewport に `viewport-fit=cover` 追加、`apple-mobile-web-app-capable` / `apple-mobile-web-app-status-bar-style` / `apple-mobile-web-app-title` メタ追加
2. **100vh → 100dvh**（フォールバック併記）: App.css L18 / Layout.css L3 / Sidebar.css L8 / ReservationListPage.css L452 / LoginPage.css L5 / calc(100vh−…) 4箇所
3. **入力ズーム対策**: input / select / textarea の font-size を 16px に統一（iPadで入力欄フォーカス時に画面が勝手にズームするのを防ぐ。既存フォームの見た目が僅かに変わるためユーザーにビフォーアフター提示）
4. 外側クリック判定 `mousedown` → `pointerdown`（ReservationCreatePage.jsx L93 / GuestSection.jsx L138 / Timeline.jsx L40。タップでの閉じ動作安定化）
5. `-webkit-tap-highlight-color: transparent` と `overscroll-behavior` の共通設定（styles/touch.css 新設 or variables.css 系に追加）

**検証**: iPadシミュレータで下端見切れ・入力ズーム消失。PCで表示変化なし。

### Phase 1: フロントモードの骨格（見積: 中）

1. ルーティング `/front` 追加（規約 #24: 固定パスの登録順に注意）
2. `FrontLayout` + 大型タブナビ + ヘッダー（3.1）
3. **PINパッドログイン**: LoginPage に大型テンキー（0-9・訂正）を追加。タッチ環境ではテンキー表示、PCでは従来のキーボード入力（認証APIは既存のまま）
4. フロント端末フラグ（localStorage）とログイン後の初期遷移
5. 本日ボードの一覧表示（3.2。カードUIとポーリング）
6. PWA調整: manifest の `start_url` 検討（フロント端末は `/front` 起動にしたいが、PC/タブレットで manifest は共用のため、**start_url は `.` のまま**とし、ログイン後のフラグ遷移で解決する方式を推奨）

### Phase 2: チェックインフロー（見積: 中）

3.3 の実装。バックエンド変更なしを原則とし、既存APIで不足が出たら都度ユーザーに相談。

### Phase 3: チェックアウト・精算フロー（見積: 中〜大）

3.4 の実装。**iPadでのPDF（領収書）表示・AirPrint が最大の検証ポイント**。standalone PWA でPDFの新規タブが開けない場合は、Safari側で開く・画面内プレビュー等の代替をユーザーと相談。

### Phase 4: 候補機能と磨き込み（見積: 中。着手前に採否をユーザーと決める）

1. 3.5 の候補機能（POS / 部屋状況閲覧 / パスポート撮影）
2. ホーム画面アイコン: coral-icon.svg から 180px apple-touch-icon と maskable 512px を生成（`rsvg-convert`。規約 #17）。**public/ 直下の差し替えは Cloudflare エッジキャッシュに注意（規約 #23）— deploy.sh の自動パージ対象を確認**

4. 縦持ちレイアウトの最終調整

### Phase 5: 検証 → 本番デプロイ

1. **検証マトリクス**（iPadシミュレータ + 可能なら実機）
   - 横・縦 × Safari・ホーム画面PWA の4通り

3. 実装記録を `docs/iPad対応_実装記録.md` に残し、知見を規約ファイルへ追記

## 5. リスクと注意（Opus 4.8 必読）

1. **iPad standalone PWA での PDF 表示/印刷が最大の技術リスク**（Phase 3）。早めに実機/シミュレータでスパイク検証すること
2. Phase 0 の入力16px化・dvh化は**全画面に影響**する。PC回帰確認を怠らない
3. フロントモードは新規UIだが、**ビジネスロジックは既存API・既存規約に完全に従う**: 楽観ロック（#16）、useConfirm（#14）、日付ユーティリティ（#2 / utils/date.js）、共通定数（#15 / constants.js）、goods行の扱い（#25）、TL原本フィールド不変（#12）
4. 未アサイン予約のCIブロックは、CheckinController 側の既存バリデーションを確認してからUI文言を作る（フロント側だけで判定しない）
5. ポーリング更新中にスタッフが精算操作している場合の画面上書きに注意（編集中はポーリング一時停止等、既存画面の作法に合わせる）
6. 候補機能（3.5）とタップ挙動の細部は**実装前にユーザーに確認**する

## 付録A: 全画面タッチ対応調査（2026-07-24 Fable 5。将来の参考資料）

棲み分け方針により対応不要になったが、**将来「iPadでアサインもやりたい」となった場合の見積り資料**として残す。

**根本課題: 既存コードベースはデスクトップ（マウス）専用前提。** `touch-action` 0件、`dvh` 0件、`matchMedia` 0件、タブレット帯メディアクエリ 0件。

### タッチで完全に動かないもの（対応する場合はPointer Events化が必要）

| 箇所 | 実装 | 問題 |
|------|------|------|
| アサインボード D&D | AssignBoardPage.jsx L388-392, 427-429, 559-564, 613-615 | HTML5 DnD API は iOS Safari のタッチで発火しない |
| バーのリサイズ（延泊/短縮） | 同 L522-550, 585 | mousemove/mouseup のみ。ハンドル幅8px |
| バーの操作メニュー | 同 L571-577, 621 | onContextMenu（右クリック）専用 |
| マスタ並び替え共通フック | hooks/useDragReorder.js 全体 | HTML5 DnD。設定4画面（PaymentMethod/Product/Room/RoomType）が対象 |
| 部屋インジケーター編集 | RoomIndicatorPage.jsx L270-334 | HTML5 DnD |
| サイドバー折りたたみ | Sidebar.css L147-178 | 幅6px＋hover時のみアイコン表示 |

### その他の記録

- hover依存の編集アイコン: ReservationDetailPage.css L35/46、GuestDetailPage.css L403/412
- 小ターゲット: チェックボックス14px（ReservationListPage.css L237）、数量ボタン24px（ProductPicker.css L171）、小型ボタン高さ20〜30px多数
- モーダル固定幅: ReservationCreatePage.css L460（min-width:380px）、SettingsPage.css L244（400px）
- 在庫表は実質幅 約2760px の横スクロール前提（RoomInventoryPage.css L75、sticky左列あり）
- 対応する場合の技術方針（当時の結論）: 新規ライブラリは入れず Pointer Events + `setPointerCapture` で自前統一。「タップ=従来動作 / 長押し250ms=ドラッグ開始 / 先に指が動いたらスクロール」の判別、エッジオートスクロール、`touch-action` 制御、規約 #7/#8/#9 の資産は維持
