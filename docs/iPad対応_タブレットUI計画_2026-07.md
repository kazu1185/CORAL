## 1. 前提と目標

### ターゲット環境

| 項目 | 値 |
|------|-----|
| デバイス | iPad Pro 13インチ（M4）。論理解像度 **横1376×1032pt / 縦1032×1376pt** |
| ブラウザ | Safari（iPadOS）＋ ホーム画面追加（PWA standalone）を正式サポート |
| 操作 | タッチ（指）が主。Apple Pencil・トラックパッドでも自然に動くこと |
| 併存環境 | 既存のPC（マウス）運用はそのまま維持。同一コードで両対応 |

### 目標「超一流」の定義

1. **全業務がタッチだけで完結する**（現状: アサイン・延泊短縮・並び替え・バー操作メニューがタッチで完全に不可能）
2. **44pt基準のタップターゲット**（Apple HIG準拠）。押し間違いが起きない
3. **ドラッグ操作が気持ちいい**: 長押しでバーが持ち上がる、端に近づくと自動スクロール、ゴースト表示
4. **iPadネイティブアプリと見紛う挙動**: 入力時の勝手なズームなし、アドレスバーによる下端見切れなし、ラバーバンドの暴れなし、ホーム画面起動でフルスクリーン
5. **Patinaデザイン言語の維持**: variables.css のトークンのみ使用。ハードコード色禁止（規約 #21）

### スコープ外

- 清掃スタッフ用画面（`/housekeeping`）は現在プレースホルダ（App.js L46）。**本計画では作らない**。Phase 13 で新規実装する際に、本計画で整備するタッチ基盤の上に最初からタッチ前提で設計する
- スマホ（〜767px）対応。タブレット＋デスクトップに集中する

## 3. 技術方針（重要な設計判断）

### 3.1 D&D は Pointer Events で自前統一（新規ライブラリは入れない）

- **HTML5 DnD API は全廃**し、`onPointerDown / onPointerMove / onPointerUp` + `setPointerCapture()` で実装する
- 理由:
  - Pointer Events はマウス・タッチ・Pencil を1系統で扱える（分岐コード不要）
  - アサインボードのD&Dは「セル単位スナップ＋泊数バー＋確認ダイアログ」の独自ロジックで、dnd-kit 等を入れても結局カスタム実装になる。依存を増やすリスク（CRAのバンドル・保守）に見合わない
  - 規約 #7（バー幅はCSS calc）・#8（全ドラッグに確認ダイアログ）・#9（リサイズ後クリック抑制）の既存資産をそのまま活かせる
- 共通化: **`hooks/usePointerDrag.js` を新設**し、「長押し検知 → ドラッグ開始 → 移動中コールバック → ドロップ」の骨格を1箇所に置く。AssignBoard・RoomIndicator・useDragReorder すべてこれを土台にする

### 3.2 スクロールとドラッグの共存ルール（タッチの肝）

タッチでは「スクロールしたいのか、ドラッグしたいのか」の判別が必要。以下で統一する:

| 操作 | 挙動 |
|------|------|
| バー/カードをタップ | 従来どおり（詳細を開く等） |
| バー/カードを**長押し（250ms）** | ドラッグモード開始。バーが少し拡大＋影（持ち上がり演出）、以降指に追従 |
| 長押し前に指が動いた（±8px超） | 通常スクロールとして続行（ドラッグ開始しない） |
| ドラッグ中に画面端へ接近 | ボード/リストを自動スクロール（エッジオートスクロール） |
| マウスの場合 | 長押し不要。従来どおり即ドラッグ開始（`e.pointerType === 'mouse'` で分岐） |

- ドラッグ可能要素には `touch-action: none` をドラッグ開始後に適用（平常時は `pan-x pan-y` でスクロールを殺さない）
- `-webkit-touch-callout: none` と `user-select: none` をドラッグ対象に明示（長押しでiOSのコールアウトが出るのを防ぐ）
- 規約 #9 の「リサイズ後100msクリック抑制」の考え方をドラッグ全般に適用（pointerup 後の click 抑制）

### 3.3 hover 依存UIの原則

- CSS は `@media (hover: hover)` / `@media (hover: none)` で分岐する
- **タッチ環境では「hoverで出るもの」を常時表示にする**（編集アイコン等）。タップ2回要求のUIにはしない
- ツールチップ（アサインボードのバー詳細）はタッチでは出さなくてよい（タップで詳細が開けるため）

### 3.4 レイアウト戦略

- 13インチ横（実効幅 約1376pt − サイドバー232px = 1144px）は現行デスクトップレイアウトがほぼそのまま成立する。**大改造はせず、ブレークポイントで微調整**する
- 新設ブレークポイント: `@media (max-width: 1140px)`（iPad縦 1032pt を包含）— サイドバーを自動で折りたたみ60px化、モーダル幅を `min(400px, calc(100vw - 32px))` 化
- 横スクロールテーブル（在庫30日表・一覧系）は sticky 列＋慣性スクロールで現状維持。`overscroll-behavior: contain` を追加してラバーバンドの連鎖を防ぐ

### 3.5 PWA（ホーム画面追加）を正式な起動方法にする

- standalone 起動ならアドレスバー問題が消え、フルスクリーンでネイティブ感が最大化する
- Safariブラウザ起動でも破綻しないよう dvh / セーフエリア対応は行う（PWAはあくまで推奨形態）

## 4. 実装フェーズ詳細

### Phase 0: 土台（見積: 小。ただし全フェーズの前提）

1. **index.html**
   - viewport を `width=device-width, initial-scale=1, viewport-fit=cover` に変更
   - `<meta name="apple-mobile-web-app-capable" content="yes">`、`<meta name="apple-mobile-web-app-status-bar-style" content="default">`、`<meta name="apple-mobile-web-app-title" content="CORAL PMS">` を追加
   - apple-touch-icon はまず既存 logo192.png のままでよい（Phase 4 で180px専用アイコンを coral-icon.svg から生成）
2. **100vh → 100dvh**（フォールバック併記: `height: 100vh; height: 100dvh;`）
   - App.css L18 / Layout.css L3 / Sidebar.css L8 / ReservationListPage.css L452 / LoginPage.css L5 / `calc(100vh - …)` 4箇所
3. **セーフエリア**: Layout / Sidebar / 固定オーバーレイに `env(safe-area-inset-*)` の padding を追加（standalone起動時のホームインジケータ回避）
4. **入力ズーム対策**: input / select / textarea の `font-size` を **16px** に統一する共通ルールを variables.css 系に追加（本文15pxは維持。規約 #21 の下限ルールと矛盾しない）
5. **共通タッチCSS**: `styles/touch.css` 新設 — `@media (hover: none)` 用の共通調整、`-webkit-tap-highlight-color: transparent`、`overscroll-behavior` 等。App.js でグローバルインポート
6. 外側クリック判定 `mousedown` → `pointerdown`（ReservationCreatePage.jsx L93 / GuestSection.jsx L138 / Timeline.jsx L40）

**検証**: iPadシミュレータのSafariで下端見切れ・入力ズームが消えたこと。PCで従来表示に変化がないこと。

### Phase 1: アサインボードのタッチ対応（本丸。見積: 大）

対象: `pages/AssignBoardPage.jsx` / `.css`

1. **`hooks/usePointerDrag.js` 新設**（3.2 の骨格。長押し検知・移動閾値・エッジオートスクロール・click抑制を内包）
2. **バーのD&D移動 / 未アサインカードのアサイン** を HTML5 DnD から usePointerDrag へ置換
   - ドロップ先セル判定は `document.elementFromPoint()` ＋ セルの data属性（date / room_id）で行う（ネイティブDnDのonDropに相当）
   - ドラッグ中はバーの半透明ゴーストを指に追従表示。ドロップ候補セルをハイライト
   - **確認ダイアログ（規約 #8）と CI後ブロック（規約 #5）のロジックは一切変更しない**
3. **リサイズ（延泊/短縮）** を pointer events 化
   - ハンドルを視覚幅12px・**タッチ判定領域44×44pt**（透明な ::after 拡張 or 別要素）に拡大
   - バー幅は引き続き CSS calc（規約 #7）。ドラッグ中のプレビューも泊数スナップで表現
4. **操作メニュー**: 右クリック（onContextMenu）は維持しつつ、**タップでメニューを開く導線を追加**
   - バーをタップ → 従来は即詳細遷移だった場合、タッチではまず小さなアクションシート（詳細を開く / 部屋移動 / 延泊・短縮 / …既存contextメニューと同項目）を表示する方式を推奨。ただし既存タップ挙動の変更になるため、**実装前にユーザーに「タップ=メニュー」か「タップ=詳細のまま＋長押し=メニュー」かを確認する**
5. ツールチップ（onMouseEnter）は `@media (hover: hover)` 相当の条件（pointerType判定）でマウス時のみ維持

**検証**: シミュレータ＋実機で「未アサイン→部屋」「部屋間移動」「延泊・短縮」「メニュー操作」が指だけで完結。PCマウスで従来操作の全パターンに退行がないこと（規約 #9 のクリック抑制含む）。

### Phase 2: その他のD&D・hover依存UI（見積: 中)

1. **useDragReorder.js を usePointerDrag ベースに書き換え**（インターフェースは現状維持: dragId / handleDragStart 等の呼び出し側4画面の変更を最小化）
   - 保険として各行に ▲▼ 上下移動ボタンも追加する（並び替えはD&Dが失敗しても業務が止まらないように）
2. **RoomIndicatorPage 編集モードD&D** を同様に置換
3. **サイドバー折りたたみ**: 6px hover トグルを廃止し、**Header に常設の折りたたみボタン（44pt）** を新設。既存トグルはマウス環境（hover: hover）でのみ残す
4. hover でしか出ない編集アイコンを `@media (hover: none)` で常時表示化（ReservationDetailPage / GuestDetailPage）
5. CalendarPicker: タップ1点目選択後に「開始日選択済み」の視覚状態を追加（hoverプレビューの代替）

### Phase 3: タップターゲット・レイアウト最適化（見積: 中）

1. **44pt 基準の適用**（見た目サイズは据え置きでも、判定領域を padding / ::after で拡張する方式を基本とする）
   - チェックボックス（ReservationListPage 14px）、ProductPicker 数量ボタン（24px）、高さ20〜30pxの小型ボタン群、Header のボタン（30px）
   - 一覧テーブルの行タップ（行全体をタップ領域に。既にそうなっている画面は確認のみ）
2. **ブレークポイント `max-width: 1140px`** 新設
   - サイドバー自動折りたたみ（縦持ち時）
   - モーダル幅の可変化（ReservationCreatePage.css L460 / SettingsPage.css L244）
   - 余白・グリッドの微調整(ダッシュボードのカード列数等、実機を見て判断)
3. `overscroll-behavior: contain` を横スクロール領域（在庫表・アサインボード・一覧）に適用

### Phase 4: 超一流の仕上げ（見積: 中）

1. **ログインのPINパッド**: iPadではソフトキーボードでなく、**画面内の大型テンキー（0-9）** でPIN入力できるUIを追加（フロント共用iPadでの運用が格段に速くなる）。PCでは従来どおりキーボード入力可
2. **アクションシートの統一**: Phase 1 で作るバー操作メニューを共通コンポーネント化し、タッチ環境のコンテキストメニュー全般に使う（Patinaトーン: カード#fff・ボーダー#E6E1D6・角丸・下からのシートまたはポップオーバー）
3. **アイコン刷新**: coral-icon.svg から 180px apple-touch-icon / 512px maskable アイコンを生成（`rsvg-convert` 使用。規約 #17 参照）。manifest.json に maskable 追加。**public/ 直下の差し替えは Cloudflare キャッシュ注意（規約 #23）— deploy.sh の自動パージ対象を確認**
4. ドラッグ演出の磨き込み: 持ち上がりアニメーション（transform: scale(1.03) + shadow）、ドロップ時のスナップ、`prefers-reduced-motion` 尊重
5. 日付入力: iOSネイティブの type=date ホイールで運用に支障が出る画面があれば、既存 CalendarPicker への置き換えを検討（実機確認後にユーザーと相談）

### Phase 5: 検証 → 本番デプロイ

1. **検証マトリクス**（シミュレータ: iPad Pro 13" / 可能なら実機）
   - 横・縦の両向き × Safari・ホーム画面PWA の4通り
   - 業務シナリオ: ログイン(PINパッド) → ダッシュボード → 予約一覧(フィルタ・行タップ) → アサインボード(新規アサイン・部屋移動・延泊・短縮・メニュー) → 予約詳細(CI/CO・明細編集・領収書) → 物販POS → 在庫表 → 設定(並び替え)
   - PC回帰: 上記と同じシナリオをマウスで(既存操作に一切の退行なし)
2. デプロイ: `./deploy/deploy.sh`（config.local.php 厳守）。public/ 直下アセットを触った場合は Cloudflare パージ確認
3. 完了後、実装記録を `docs/iPad対応_実装記録.md` に残し、新たな知見を規約ファイルへ追記

## 5. リスクと注意（Opus 4.8 必読）

1. **タップ=詳細遷移の既存挙動を変えるかは未確定**（Phase 1-4）。実装前に必ずユーザーに確認する
2. **Pointer Events 化で最も壊れやすいのはPCマウス操作の回帰**。特にアサインボードの「リサイズ後クリック抑制」（規約 #9）・ドラッグ確認ダイアログ（規約 #8）は必ず回帰確認する
3. iOS Safari は `pointermove` 中に予期せぬ `pointercancel` を投げることがある（スクロール判定に負けた時）。`touch-action` の設定漏れが原因のことが多い。ドラッグ開始後の要素に `touch-action: none` が効いているか確認
4. `useDragReorder` の呼び出し側4画面はインターフェース互換で乗り切る。呼び出し側の大改造はしない
5. 入力フォント16px化で既存フォームの見た目が僅かに変わる。ユーザーにビフォーアフターを見せる
6. 規約 #14: 確認UIは必ず `useConfirm`。アクションシート新設時もブラウザ標準ダイアログは使わない
7. CLAUDE.md 絶対ルール1: **各フェーズの実装着手前にユーザーの承認を得る**（この計画書の存在をもって全フェーズ承認済みとはみなさない）

## 6. 推奨実装順序と規模感

| フェーズ | 規模 | 依存 |
|---------|------|------|
| Phase 0 | 小（半日相当） | なし。最初に必ずやる |
| Phase 1 | 大（アサインボードが本丸） | Phase 0 |
| Phase 2 | 中 | Phase 1 の usePointerDrag |
| Phase 3 | 中 | Phase 0 |
| Phase 4 | 中 | Phase 1〜3 |
| Phase 5 | 検証＋デプロイ | 全部 |

Phase 0 → 1 → 2 → 3 → 4 → 5 の直列を推奨。Phase 3 は Phase 1・2 と並行可能だが、コンフリクト回避のため直列が無難。
